# pulse_inspector_vcd (`PulseInspectorVcd`)

ESPHome-компонент, который превращает ESP32 в **сетевой логический анализатор**.
Подписывается на каналы родительского
[`pulse_inspector`](../pulse_inspector/README.md) и отдаёт собранные фронты
сигналов как живой поток в формате
**[VCD (Value Change Dump)](https://en.wikipedia.org/wiki/Value_change_dump)**
по обычному TCP-порту.

English version: [README.en.md](README.en.md).

| Метаданные ESPHome | |
|---|---|
| `CODEOWNERS` | `@anat0l` |
| `DEPENDENCIES` | `esp32`, `pulse_inspector` |
| `MULTI_CONF` | `false` |

Индекс всех компонентов: [`../README.md`](../README.md).

К результату можно подключиться любым TCP-клиентом
(`nc`/`ncat`/`socat`/PulseView), сохранить в файл или открыть «вживую»
в [PulseView](https://sigrok.org/wiki/PulseView) или
[GTKWave](https://gtkwave.sourceforge.net/) — там же на запись можно
повесить готовые декодеры протоколов (UART, I²C, SPI, и т.п.).

> Зачем это нужно: если у вас слабый или редкий сигнал, на который
> неудобно «вешаться» отдельным логическим анализатором (далеко от
> рабочего места, шумная USB-земля, нет провода нужной длины), ESP32
> запросто заменит лог-анализатор на 2–4 канала и отдаст всё по Wi-Fi.

## Возможности

- Захват **всех каналов** родительского `pulse_inspector`
  одновременно (до 94 каналов на один экспортёр; ограничение формата
  VCD на однобайтовые идентификаторы).
- Двухэтапная буферизация:
  1. **Кольцевой буфер** последних `buffer_size` фронтов — отдаёт
     «pre-trigger» историю клиенту сразу после подключения.
  2. **Очередь живого потока** — каждое новое изменение GPIO попадает
     в TCP-сокет, пока клиент подключён.
- Корректный заголовок VCD: `$timescale`, `$var` для каждого канала с
  именем `chN_gpioM`, начальные уровни (`0`/`1`/`x`), монотонные
  временные метки.
- **Сетевая устойчивость**:
  - lwIP TCP keepalive — определяет «полу-мёртвый» сокет за десятки
    секунд, а не за 1–3 минуты типичного таймаута ретрансмиссий.
  - Application-level heartbeat (`$comment` каждые N секунд) — не даёт
    NAT/AP сбрасывать «бездействующее» соединение.
  - Настраиваемый `SO_SNDTIMEO` для отключения зависших клиентов.
- Не блокирует ISR: в горячем пути только `xQueueSend`/`memcpy`;
  все «тяжёлые» операции (форматирование строк, `send()`) идут в
  отдельной FreeRTOS-задаче.

## Что такое VCD

**Value Change Dump** — текстовый ASCII-формат для записи изменений
цифровых сигналов во времени. Часть стандарта IEEE 1364 (Verilog),
де-факто универсальный формат логических анализаторов.

Идея простая:

1. Заголовок описывает временной масштаб и список «проводов».
2. Дальше пишутся **только моменты, когда что-то изменилось**:
   маркер времени `#1234` плюс новое значение бита, например `1!`.

Для разреженных линий (UART/MDB/I²C) это очень компактно — типичный
байт UART даёт 4–10 транзакций × ~15 байт VCD-текста = ~60–150 байт,
тогда как сырой PCM на 1 МГц забил бы канал.

VCD понимают:

- **PulseView** (sigrok) — бесплатный, с большим набором декодеров.
- **GTKWave** — классический viewer, идёт со многими дистрибутивами.
- **Verilator / Icarus / ModelSim** — RTL-симуляторы.

## Архитектура

```
       ┌──────────────────────────────────────────────────────────┐
       │  pulse_inspector (родитель)                           │
       │  ┌────────────────┐    ┌────────────────┐                │
       │  │ Channel 0 ISR  │    │ Channel 1 ISR  │   ...          │
       │  └───────┬────────┘    └───────┬────────┘                │
       └──────────┼─────────────────────┼─────────────────────────┘
                  │ pulse callback      │
                  ▼                     ▼
       ┌──────────────────────────────────────────────────────────┐
       │  pulse_inspector_vcd                                  │
       │  ┌──────────────────────────┐  ┌─────────────────────┐  │
       │  │ Кольцевой буфер          │  │ Очередь живого      │  │
       │  │ (последние buffer_size   │  │ потока на клиента   │  │
       │  │  фронтов всех каналов)   │  │ (live_queue_depth)  │  │
       │  └─────────────┬────────────┘  └─────────┬───────────┘  │
       │                │ snapshot                │              │
       │                └────────────┬────────────┘              │
       │                             ▼                            │
       │                ┌──────────────────────┐                  │
       │                │ TCP server task      │                  │
       │                │ (порт `port`)        │                  │
       │                │ - SO_SNDTIMEO        │                  │
       │                │ - SO_KEEPALIVE       │                  │
       │                │ - $comment heartbeat │                  │
       │                └──────────┬───────────┘                  │
       └───────────────────────────┼──────────────────────────────┘
                                   │
                                   ▼
                        ╔══════════════════════╗
                        ║ Wi-Fi / TCP клиент   ║
                        ║ ncat | PulseView ... ║
                        ╚══════════════════════╝
```

В каждый момент допускается **один** активный клиент. Следующий
`accept()` произойдёт после закрытия текущего.

## Требования

- ESP32 (любой вариант, поддерживаемый ESP-IDF).
- ESPHome с `framework: type: esp-idf`. Вариант на Arduino не
  поддерживается: используются прямые `lwip/sockets.h` и FreeRTOS API.
- Сконфигурированный родительский `pulse_inspector` с одним и более
  каналом (см. соседний компонент).
- Компонент `wifi:` (или другая сеть) — TCP-сервер поднимается с
  приоритетом `AFTER_WIFI`.

## Установка

Подключите оба компонента в `external_components`:

```yaml
external_components:
  - source:
      type: local
      path: components
    components:
      - pulse_inspector
      - pulse_inspector_vcd
```

## Минимальный пример

```yaml
pulse_inspector:
  id: inspector
  channels:
    - id: ch_a
      input_pin: GPIO16
    - id: ch_b
      input_pin: GPIO18
      invert_in: true

pulse_inspector_vcd:
  inspector_id: inspector
```

Запустить захват:

```bash
ncat <ip-устройства> 9000 > capture.vcd
```

…работать с автоматом / схемой нужное время, нажать `Ctrl+C` —
получится `capture.vcd`, который открывается в PulseView/GTKWave.

## Полный пример со всеми параметрами

```yaml
pulse_inspector_vcd:
  id: vcd
  inspector_id: inspector
  port: 9000
  buffer_size: 4096

  # Параметры устойчивости TCP-соединения и потока:
  send_timeout: 30s              # 1s..3600s
  live_queue_depth: 1024         # 64..16384 событий на клиента
  app_keepalive_interval: 15s    # 0s = выключить $comment heartbeat
  tcp_keepalive:
    enabled: true
    idle: 30s                    # тишина до первого probe
    interval: 10s                # период между probe-пакетами
    count: 3                     # сколько probe пропустить до разрыва
```

## Описание параметров

### Корневые параметры

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| `id` | id | автогенерация | ID компонента в стиле ESPHome. |
| `inspector_id` | id (`PulseInspector`) | **обязательный** | Ссылка на родительский `pulse_inspector`. Все его каналы будут попадать в VCD. |
| `port` | int (1..65535) | `9000` | TCP-порт, на котором висит сервер. На один порт можно одновременно держать только одного клиента. |
| `buffer_size` | int (128..65536) | `4096` | Размер **кольцевого буфера** фронтов. При подключении клиент сразу получает эти события как pre-trigger историю. Чем больше — тем дольше «история до подключения», но и больше RAM (8 байт на событие). |

### Параметры устойчивости соединения

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| `send_timeout` | время (1s..3600s) | `30s` | Таймаут `SO_SNDTIMEO` на клиентском сокете. Если буфер ядра не удалось «слить» за это время (медленный клиент, перегруженная сеть), `send()` возвращает ошибку и сервер закрывает соединение. Малое значение режет «зависших» клиентов быстрее, большое — терпит всплески. |
| `live_queue_depth` | int (64..16384) | `1024` | Глубина FreeRTOS-очереди, в которую пишутся события **во время активного соединения**. Если задача ISR/канала генерит фронты быстрее, чем сервер успевает их отправлять (плохой Wi-Fi), переполнение приведёт к **потере событий** (счётчик `live overflow`), но соединение останется живым. |
| `app_keepalive_interval` | время (0s..3600s) | `15s` | Интервал, через который при простое в поток шлётся строка `$comment esphome vcd keepalive $end`. VCD-вьюверы её игнорируют, но байты на проводе не дают NAT/маршрутизатору считать соединение idle и сбросить его. `0s` отключает heartbeat. |

### Подгруппа `tcp_keepalive` (lwIP TCP-уровень)

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| `enabled` | bool | `true` | Включить `SO_KEEPALIVE` на клиентском сокете. Когда выключен, «полу-мёртвый» (peer молчит, RST не пришёл) сокет ловится только пассивно — через таймаут TCP-ретрансмиссий lwIP, что обычно занимает 60–180 с. |
| `idle` | время (1s..3600s) | `30s` | Сколько секунд тишины должно пройти до **первого** probe-пакета. |
| `interval` | время (1s..600s) | `10s` | Интервал между probe-пакетами, если ответа нет. |
| `count` | int (1..20) | `3` | Сколько подряд непрошедших probe считать «клиент мёртв» и закрыть сокет. |

> Эффективное «время обнаружения мёртвого клиента» при включённых
> keepalive ≈ `idle + interval × count`. С дефолтами это **~60 секунд**.

## Использование захвата

### Рекомендуемый способ — Python-клиент (`tools/vcd_capture_tcp.py`)

В репозитории есть готовый скрипт, без буферизации и без сюрпризов
платформы:

```cmd
python tools\vcd_capture_tcp.py 192.168.1.50 9000 capture.vcd
```

Полезные опции:

```cmd
python tools\vcd_capture_tcp.py 192.168.1.50 9000 capture.vcd --duration 600
python tools\vcd_capture_tcp.py 192.168.1.50 9000 capture.vcd --idle-timeout 60
```

Скрипт раз в 5 секунд печатает в stderr строку вида
`[client] <s>s <bytes>B (<rate> B/s) chunks=N` — её удобно сверять с
`VCD stream: ... bytes sent ...` в логе ESPHome. Если цифры расходятся
— проблема в сети или приёмнике, а не в устройстве.

`Ctrl+C` корректно закрывает поток и оставляет валидный `.vcd` файл.

### Альтернатива — `ncat` / `nc` / `socat` (Linux/macOS)

```bash
ncat 192.168.1.50 9000 > capture.vcd
nc   192.168.1.50 9000 > capture.vcd
socat TCP:192.168.1.50:9000 OPEN:capture.vcd,creat,trunc
```

Файл VCD **корректен на любом моменте обрыва** (формат не требует
закрывающих строк), поэтому захват можно прерывать в любой момент и
сразу открывать.

> **Известная проблема на Windows.** Связка `cmd.exe` + `ncat` (от
> nmap) на Windows стабильно обрывает поток через несколько минут с
> размером файла ~400 КБ — у клиента переполняется внутренний буфер,
> и Windows-сокет шлёт RST в сторону ESP32 (в логе ESPHome видно
> `last_errno=104 (Connection reset by peer)`). На стороне устройства
> данные при этом отправляются корректно. Используйте `vcd_capture_tcp.py`
> — он этой проблемы не имеет. PowerShell-овский оператор `>`
> аналогично перекодирует поток через `Out-File` и подходит ещё хуже.

### Стримить и одновременно тратить трафик «на лету»

```bash
ncat 192.168.1.50 9000 | tee capture.vcd | grep --line-buffered "#"
```

### Несколько окон / параллельные клиенты

Сервер обрабатывает **только одного клиента**. Второе подключение
встанет в `accept()` и активизируется только после `close()` первого.
Если нужно несколько одновременных потоков, поднимите второй
`pulse_inspector_vcd` на другом порту, ссылающийся на тот же
`pulse_inspector` (`inspector_id: inspector`).

## Просмотр в PulseView / GTKWave

### PulseView

1. Скачайте трассу (`ncat ... > capture.vcd`).
2. В PulseView: `Open` → выберите `capture.vcd`.
3. Перетащите нужные провода (`ch0_gpio16`, `ch1_gpio18`, …) в нужный
   порядок.
4. ПКМ по «дорожке канала» → `Add protocol decoder...` → выберите
   подходящий (UART, I²C, SPI и т.п.) и укажите параметры (для UART
   обычно baudrate, число данных, parity, stop, MSB/LSB-first).

### GTKWave

`gtkwave capture.vcd`. Слева в дереве сигналов появятся `ch0_gpio16`,
`ch1_gpio18`, … — добавляйте их в правую панель и масштабируйте
обычной мышью.

### Имена сигналов

Каждому каналу `pulse_inspector` соответствует один VCD-провод,
именованный как:

```
chN_gpioM
```

где `N` — индекс канала в порядке, как они объявлены в YAML, а `M` —
номер GPIO. Идентификатор VCD (короткий печатный символ) выдаётся
автоматически, начиная с `!`.

## Диагностика

При запуске устройство печатает:

```
[C][pulse_inspector_vcd]: pulse_inspector_vcd:
[C][pulse_inspector_vcd]:   TCP port: 9000
[C][pulse_inspector_vcd]:   Buffer size: 4096 events
[C][pulse_inspector_vcd]:   Live queue depth: 1024 events
[C][pulse_inspector_vcd]:   Send timeout: 30 s
[C][pulse_inspector_vcd]:   App keepalive: 15 s
[C][pulse_inspector_vcd]:   TCP keepalive: idle=30s interval=10s count=3
[C][pulse_inspector_vcd]:   Download with:  nc <device-ip> 9000 > capture.vcd
```

Когда сервер слышен на порту:

```
[I][pulse_inspector_vcd]: Listening for VCD downloads on tcp/9000
```

При подключении клиента:

```
[I][pulse_inspector_vcd]: Client 192.168.1.20 connected, starting live VCD stream
```

При отключении (любом):

```
[I][pulse_inspector_vcd]: Client disconnected (live overflow during session: 0)
```

`live overflow during session` — счётчик событий, **выкинутых**
из-за переполнения `live_queue_depth`. Если он `> 0`, значит сеть
не успевала забирать данные.

## Решение проблем

### Соединение рвётся через 1–3 минуты

Это классический симптом таймаута TCP-ретрансмиссий lwIP на потерянных
пакетах Wi-Fi. С версии, где есть `tcp_keepalive` и
`app_keepalive_interval`, проблема устраняется автоматически.

Если всё же рвётся:

- Проверьте, что в логе видно `TCP keepalive: idle=… interval=… count=…`
  и `App keepalive: 15 s` (не «disabled») — значит фичи реально
  включены.
- Убедитесь, что **в логе ESPHome нет `WiFi disconnected`** в момент
  обрыва — если есть, проблема в качестве самой Wi-Fi-связи.
- Уменьшите `tcp_keepalive.idle` до 10–15 с, чтобы быстрее ловить
  «глухой» канал.
- Уменьшите `app_keepalive_interval` до 5–10 с — это поможет, если
  между ESP32 и клиентом стоит NAT/файрвол с очень короткой idle-таймером.

### `live overflow during session: N > 0` (теряются события)

Скорость генерации фронтов превышает скорость их сетевой выгрузки.
Варианты:

- Увеличьте `live_queue_depth` (например, до `4096` или `8192`):
  поглощает кратковременные всплески, но потребляет больше RAM
  (8 байт на событие).
- Улучшите Wi-Fi (ESP ближе к точке доступа, без перегруженного канала).
- Сократите количество каналов / уменьшите длительность активного
  трафика на наблюдаемой линии.

### `Failed to allocate live streaming queue`

Либо нет свободной кучи, либо `live_queue_depth` слишком большой.
Уменьшите глубину очереди или `buffer_size`.

### Клиент подключился, но файл получает усечённую запись

Скорее всего, клиент сам закрывает сокет «по таймауту приложения»
(некоторые версии `nc` так делают при простое в `>` редиректе на
SSD). Попробуйте явно `socat TCP:...:9000 OPEN:capture.vcd,creat,trunc`
или `ncat` от nmap.

### `t_us` после ~71 минуты

Внутреннее представление времени — `uint32_t` микросекунд, что даёт
~71 минуту до переполнения. На границе формат VCD «откатится» к нулю
и далее будет монотонно расти заново. Длинные сессии — открывайте в
PulseView порциями по сегменту до отката, либо переподключайтесь
каждые 60 минут (это сбросит `t0`).

## Ограничения и тонкости реализации

- **Один клиент на порт** одновременно (см. секцию «Использование»).
- **`buffer_size` ≠ длительность сессии**. Это только pre-trigger
  история, отдаваемая мгновенно при подключении. Сама сессия
  «бесконечна», пока вы держите TCP-соединение и хватает памяти на
  очередь живого потока.
- **Порядок событий** в кольцевом буфере не строго по времени, потому
  что callback'и каналов выполняются в **разных задачах FreeRTOS**.
  При выгрузке snapshot применяется `std::stable_sort`, и далее
  `#t`-маркеры VCD принудительно делаются монотонными
  (`if (t < last_t_emitted) t = last_t_emitted`). На очень плотном
  трафике на нескольких каналах это может привести к небольшим
  «слипшимся» в одну метку фронтам, но это укладывается в спецификацию
  VCD и корректно интерпретируется PulseView/GTKWave.
- **`PRODUCER_MUTEX_TIMEOUT = 10 ms`**: если мьютекс ring buffer'а
  не удалось взять за 10 мс из ISR-callback'а канала, **событие
  выбрасывается** (защита от deadlock'а на основной канал). Этого
  быть не должно при разумных настройках, но если видите потери —
  посмотрите, не блокируется ли поток сервера (сетевой стек, тяжёлая
  операция).
- **`STREAM_POLL_INTERVAL = 100 ms`**: с такой гранулярностью
  работает idle-логика (flush + heartbeat). Реальные события из
  очереди отправляются мгновенно, без 100-мс задержки.
- Включение `SO_KEEPALIVE` на ESP-IDF lwIP требует, чтобы в
  `sdkconfig` стоял `CONFIG_LWIP_TCP_KEEPALIVE=y` (по умолчанию `y`
  в современных ESP-IDF, ESPHome его сам включает).
